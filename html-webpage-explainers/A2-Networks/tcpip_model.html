What is it?

A Conceptual Framework for Networks

Digital networks are complex, interdependent systems. The TCP/IP model is a conceptual framework used to aid understanding, characterizing, troubleshooting, and standardizing network operations.

📦

Real-World Analogy

Think of sending a parcel. You write a letter (Application), put it in an envelope with a tracking number (Transport), address the envelope (Internet), and hand it to a courier with physical delivery instructions (Network Interface). Each step wraps the previous one.

🔄

Layered Approach

Each layer handles a specific task. Data flows top → bottom when sending (encapsulation) and bottom → top when receiving (decapsulation). Each layer only needs to talk to the layers directly above and below it.

The Model

Four Layers of TCP/IP

Click each layer to expand details about its role, key protocols, and functions.

🌐
Layer 4 · Top Layer
Application Layer
HTTP SMTP DNS DHCP SNMP

The Application layer is the interface between applications on a computer and the network. It defines protocols that applications use to exchange data over the network.

This is the only layer the end user directly interacts with. Protocols here dictate how data is formatted and presented for human-readable services like web browsing and email.

HTTP / HTTPS — Defines how web browsers request and receive web pages
SMTP — Simple Mail Transfer Protocol; handles outgoing email
DNS — Domain Name System; translates domain names (e.g. google.com) to IP addresses
DHCP — Dynamically assigns IP addresses to devices on a network
SNMP — Simple Network Management Protocol; used for managing and monitoring network devices
🚚
Layer 3
Transport Layer
TCP UDP

The Transport layer facilitates direct communication services for application processes across various hosts. It breaks data into segments and ensures reliable delivery.

The three-way handshake (SYN → SYN-ACK → ACK) establishes a reliable TCP connection before data is sent.

Segmentation — Breaks application data into smaller TCP segments
Sequencing — Numbers each segment so receiver can reassemble in correct order
Error checking — Uses checksums to detect corruption; requests retransmission if needed
Port numbers — Directs data to correct application (port 80 = HTTP, port 443 = HTTPS)
TCP vs UDP — TCP: reliable, ordered. UDP: faster but no guarantee of delivery
🗺️
Layer 2
Internet Layer
IP ICMP ARP

The Internet layer handles packetization, addressing, and routing of data across different networks to ensure it reaches its intended destination.

The primary protocol here is IP. Each TCP segment is wrapped in an IP packet containing source and destination IP addresses.

IP Addressing — Assigns logical addresses (IPv4/IPv6) to uniquely identify devices
Routing — Determines the best path for packets across multiple networks
Packetization — Encapsulates TCP segments into IP packets with headers
TTL (Time to Live) — Prevents packets from circulating indefinitely on the network
Header checksum — Error detection for the IP header itself
🔌
Layer 1 · Bottom Layer
Network Interface Layer
Ethernet 802.3 Wi-Fi 802.11 MAC

The Network Interface layer concerns itself with the physical and hardware aspects of network communication. This is where data actually hits the wire (or the air).

Network adapters (Ethernet cards, Wi-Fi cards) operate here, interfacing directly with the physical medium. Protocols include 802.3 and 802.11.

Framing — Encapsulates IP packets into Ethernet frames with MAC addresses
MAC Addressing — Hardware addresses that identify devices on the local network
Signal encoding — Converts frames into electrical signals, light pulses, or radio waves
Physical transmission — Sends signals over copper, fibre-optic, or wireless media
Error detection — Frame checksum (CRC) to detect transmission errors

Step-by-Step

Fetching a Web Page: All 8 Steps

A detailed walkthrough of what happens when you type a URL and press Enter.

1
Application Layer

User Requests a Web Page

The user enters a URL. The web browser (application) constructs an HTTP GET request according to the HTTP protocol. This request is passed down to the Transport layer.

2
Transport Layer

Preparing the Request for Transmission

The HTTP request becomes a TCP segment with a header containing sequencing info, source/destination port numbers (port 80 for HTTP), and a checksum. A three-way handshake (SYN → SYN-ACK → ACK) establishes a reliable connection before data transfer.

3
Internet Layer

Preparing the Segment for Routing

The TCP segment is encapsulated into an IP packet with source IP, destination IP, protocol type, TTL, and header checksum. A next-hop address (usually your router) is assigned.

4
Network Interface Layer

Final Encapsulation & Converting to Signal

The IP packet is wrapped in an Ethernet frame with destination MAC address, source MAC, and a frame checksum (FCS). The NIC converts this into electrical signals or radio waves and transmits over the physical medium.

5
Network Interface Layer (Server)

Receiving the Request on the Server

The server's NIC receives signals and converts them back to an Ethernet frame. The frame checksum is verified. If the destination MAC matches, the Ethernet header is stripped and the IP packet is passed up. The frame is processed bottom-up through all layers.

6
Application Layer (Server)

Server Responds with the Web Page

The web server generates an HTTP response with the requested HTML. This travels back down through all four layers — Application → Transport → Internet → Network Interface — then is transmitted back to the client.

7
Application Layer (Client)

User Receives the Web Page

The response travels up through the client's layers. The application layer's web browser interprets the HTML content and renders the web page for the user.

8
Transport Layer

Connection Termination

The TCP connection is gracefully closed via a FIN / ACK exchange. A FIN packet signals that the sender has finished; an ACK confirms receipt. This cleanly terminates the connection in both directions.
